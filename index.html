Базовый URL. https://mobile.5lb.pro/api/v1

1. Общие принципы

Формат успешных ответов

Auth-эндпоинты исторически отдают «голый» JSON с user, accessToken и refreshToken на верхнем уровне — это байт-в-байт совпадает с мобильным клиентом 5LB и менять формат нельзя.

Новые эндпоинты веб-кабинета (списание бонусов, статус заказа, refund) используют единый конверт:

{
  "status": "success",
  "data": { ... }
}

Формат ошибок

Веб-кабинет:

{
  "status": "error",
  "code": "CARD_NOT_OWNED",
  "message": "Bonus card not found or does not belong to user"
}

Auth (для совместимости с мобайлом):

{
  "success": false,
  "error": "CODE_INVALID",
  "attemptsLeft": 2
}

HTTP-заголовки

ЗаголовокКогдаОписание
Content-Type: application/jsonВсе POST/PATCHБэк не принимает application/x-www-form-urlencoded.
Authorization: Bearer <accessToken>Эндпоинты user JWTJWT, полученный из любого auth-эндпоинта. TTL — 15 минут для refresh-флоу, permanent для phone/email OTP. Альтернатива — cookie accessToken (см. ниже).
X-Service-Key: <secret>Эндпоинты service keyСекретный ключ Bitrix-бэка для server-to-server. Никогда не отдавать на фронт и не светить в JS. Получить у 5LB-команды.
X-City-Code: khabarovskОпционально, GET /bonuses/cardsФильтр по городу. Значения: khabarovsk, vladivostok, blagoveshensk, komsomolsk и т. п.

Cookie-режим (опционально)

Если SPA-кабинет хостится на https://app.5lb.pro или другом домене *.5lb.pro, бэкенд может отдавать accessToken / refreshToken в HttpOnly Set-Cookie. Endpoint /auth/refresh уже умеет читать токен из cookie. Для Bitrix-сайта на 5lb.pro это самый безопасный режим (XSS не доберётся до токена). Подробнее — раздел «Безопасность».

CORS

Браузер пускает запросы с домена сайта только если origin прописан в allowlist бэкенда. Перед интеграцией передайте 5LB-команде точные домены — например:

https://5lb.pro
https://www.5lb.pro
https://stage.5lb.pro

Без этого браузер ответит CORS error ещё до того, как запрос дойдёт до бэка.

2. Авторизация — обзор каналов

В кабинете поддерживаются 5 способов входа. Все они в итоге выдают accessToken и (опционально) refreshToken, после которых можно дёргать защищённые эндпоинты профиля и бонусов.

КаналЭндпоинтRefresh-токенNote
SMS-кодPOST /auth/phone/request → POST /auth/phone/verifyНет (permanent JWT)Оператор MTS Exolve. Если у юзера установлен Telegram — fallback на отправку кода через TG-бот.
Email-кодPOST /auth/email/request → POST /auth/email/verifyНет (permanent JWT)SMTP-доставка кода.
Telegram / MAXGET /auth/telegram/seamless/init → polling /auth/telegram/seamless/check/:tokenДа (30 дней)Универсальный «войти через бота» — работает и для Telegram, и для MAX. См. ниже.
VK IDGET /auth/vk/state → POST /auth/vk/exchangeДаOAuth2 с PKCE.
Apple Sign InPOST /auth/apple/verifyДаApple JS на сайте отдаёт identityToken — бэк проверяет подпись.
Что НЕ дают авторизованные эндпоинты: токен не привязан к IP — кража токена (XSS) равносильна угону аккаунта. Используйте HttpOnly cookie или храните access-токен только в памяти JS (не в localStorage), refresh-токен — только в HttpOnly cookie. Бэк сам ставит cookie при auth-ответе для доменов *.5lb.pro.

3. Авторизация по SMS

POST /auth/phone/request public

Запрашивает отправку 4-значного OTP на номер телефона. Rate-limit: 1 SMS в минуту на номер, 3 в час, 5 в сутки.

Request

POST /api/v1/auth/phone/request
Content-Type: application/json

{
  "phone": "+79141234567",
  "deviceFingerprint": "browser:chrome:linux:..."   // опционально, для anti-fraud
}

Response 200

{
  "success": true,
  "requestId": "01HJZ5...",
  "expiresAt": "2026-05-20T12:34:56.000Z",
  "channel": "sms",        // "sms" | "telegram" (если у юзера есть TG — бот может отправить код)
  "maskedDestination": "+79141***567"
}

Возможные ошибки

HTTPcode/errorЧто показать
400INVALID_PHONE«Неверный формат номера»
429rate_limited + Retry-After: 60«Повторите через N сек.»
503SMS_PROVIDER_UNAVAILABLE«Не удалось отправить SMS, попробуйте позже»

POST /auth/phone/verify public

Проверяет OTP и выдаёт permanent JWT. 5 попыток, потом код блокируется. Создаёт юзера при первом входе.

Request

POST /api/v1/auth/phone/verify
Content-Type: application/json

{
  "phone": "+79141234567",
  "code":  "1234",
  "deviceName": "Bitrix Web",
  "appVersion": "1.0.0",
  "platform":   "web",
  "referralCode": "ABC123"     // опционально (приведённый код реферала)
}

Response 200

{
  "success": true,
  "accessToken": "eyJhbGciOiJIUzI1NiIs...",
  "refreshToken": null,                       // SMS-OTP без refresh
  "isNewUser": false,
  "user": {
    "id": "ckxyz...",
    "phone": "+79141234567",
    "email": null,
    "firstName": "Никита",
    "lastName": "Тер.",
    "displayName": "Никита Т.",
    "bonusBalance": 1250,
    "currentCityCode": "khabarovsk",
    "isRegistrationComplete": true,
    "bonusCards": [ /* см. п.10 */ ],
    "addresses": [ ... ],
    "referrals": [ ... ],
    "createdAt": "2024-09-12T10:11:22.000Z"
  }
}

Ошибки

HTTPerrorЧто значит
400MISSING_CODE, CODE_INVALID_FORMAT, CODE_NOT_FOUNDОпечатка в форме / код просрочен.
403CODE_INVALID + attemptsLeftНеверный код, показать счётчик попыток.
403CODE_INVALID_LOCKEDПопытки исчерпаны, запросить новый код.
410CODE_EXPIREDПросрочен — запросить новый.

5. Авторизация через Telegram / MAX (seamless)

Один универсальный flow для двух мессенджеров: фронт получает временный токен и две deep-link'и (в Telegram-бота и в MAX-бота). Юзер открывает любую — бот авторизует на сервере, фронт поллит статус.

Шаг 1. GET /auth/telegram/seamless/init

GET /api/v1/auth/telegram/seamless/init

→ 200 OK
{
  "authToken":           "abc123def456...",          // непредсказуемый opaque token
  "telegramDeepLink":    "https://t.me/lb5_bot?start=auth_abc123def456",
  "telegramBotUsername": "lb5_bot",
  "maxDeepLink":         "https://max.ru/lb5_max_bot?start=auth_abc123def456",
  "maxBotUsername":      "lb5_max_bot",
  "expiresAt":           "2026-05-20T13:00:00.000Z"  // 10 минут
}

Сайт показывает 2 кнопки: «Войти через Telegram» (открывает telegramDeepLink в новом окне) и «Войти через MAX» (maxDeepLink). После клика начинаем polling.

Шаг 2. GET /auth/telegram/seamless/check/:authToken

Polling. Рекомендуемый интервал — 2 секунды. До 5 минут (потом — токен просрочен).

GET /api/v1/auth/telegram/seamless/check/abc123def456...

→ 200 OK  (юзер ещё не открыл бота)
{ "status": "pending" }

→ 200 OK  (юзер открыл и нажал Start)
{
  "status": "completed",
  "accessToken":  "eyJhbGci...",
  "refreshToken": "f5e7c9a2...",   // 30 дней
  "user": { /* полный UserDto */ }
}

→ 410 Gone (просрочен)
{ "status": "expired" }
Один flow — два мессенджера. Бэк сам определяет, откуда юзер пришёл (TG или MAX) по callback бота. Если у юзера уже привязан номер телефона, оба мессенджера сольются в один аккаунт автоматически.

6. Авторизация через VK ID

Шаг 1. GET /auth/vk/state

Получаем state (CSRF) и code_challenge (PKCE) — нужны для редиректа в VK.

GET /api/v1/auth/vk/state
→ 200 OK
{
  "state":         "01HJZ...",
  "codeChallenge": "abc123def...",
  "codeChallengeMethod": "S256",
  "redirectUri":   "https://5lb.pro/auth/vk/callback"
}

Фронт делает редирект:

https://id.vk.com/authorize?
  response_type=code&
  client_id=<VK_APP_ID>&
  redirect_uri=https://5lb.pro/auth/vk/callback&
  state=<state>&
  code_challenge=<codeChallenge>&
  code_challenge_method=S256&
  scope=email

Шаг 2. POST /auth/vk/exchange

На callback-странице (/auth/vk/callback?code=…&state=…) фронт отправляет:

POST /api/v1/auth/vk/exchange
Content-Type: application/json

{
  "code":         "1234567890abc",
  "state":        "01HJZ...",       // тот же, что получили из /vk/state
  "deviceId":     "<device_id из VK>"
}

→ 200 OK
{
  "success": true,
  "accessToken":  "eyJ...",
  "refreshToken": "f5e7c9a2...",
  "user": { ... }
}

  "user": { ... }

8. Обновление токенов и logout

POST /auth/refresh

Используется для Telegram/MAX/VK/Apple (где есть refreshToken). SMS/Email — permanent JWT, refresh не нужен.

POST /api/v1/auth/refresh
Content-Type: application/json

{
  "refreshToken": "f5e7c9a2..."
}

# либо пустое тело — токен возьмётся из cookie `refreshToken`

→ 200 OK
{
  "accessToken":  "новый JWT",
  "refreshToken": "новый refresh (ротация)",
  "user":         { ... }
}

Rate-limit: 5 попыток / 60 сек на IP. Старый refresh после ротации становится невалидным.

POST /auth/logout user JWT

POST /api/v1/auth/logout
Authorization: Bearer <accessToken>

→ 200 OK
{ "success": true }

GET /auth/me user JWT

Тестовый эндпоинт «кто я» — возвращает полный UserDto (тот же, что в auth-ответах). Удобно для проверки валидности токена и подгрузки данных в шапку кабинета.

10. Бонусные карты — список и балансы

Возвращает все активные бонусные карты юзера с балансами и метаданными. Если юзер ни разу не получал карту в текущем городе — будет lazy-создана BASE-карта.

→ 200 OK

11. Заказы на сайте: списание бонусов

Это новый блок endpoint'ов, специально для Bitrix. Идея: при оформлении заказа на сайте юзер выбирает бонусную карту и сколько бонусов списать. Bitrix-бэк дёргает наш бэк, который атомарно списывает бонусы и пишет «фискальную» запись заказа. При отмене заказа — refund.

Eager-deduct. Бонусы списываются сразу при создании заказа. Если заказ отменён или товар возвращён — Bitrix-бэк вызывает refund, мы возвращаем бонусы на ту же карту. Транзакционно, идемпотентно по externalId.

11.1 POST /web/orders/quote user JWT

Read-only превью: «сколько максимум можно списать с этой карты, если чек на X рублей и юзер хочет Y бонусов». Применяет правило maxRedemptionPercent и обрезает по балансу. Ничего не пишет в БД.

Request

POST /api/v1/web/orders/quote
Authorization: Bearer <accessToken>
Content-Type: application/json

{
  "userCardId":     "ckxyz-abc123",
  "totalSumRub":    2500.00,
  "requestedBonus": 1000
}

Response 200

{
  "status": "success",
  "data": {
    "userCardId":              "ckxyz-abc123",
    "cardCode":                "BASE",
    "cardName":                "Базовая карта 5LB",
    "cardBalance":             1250,
    "maxRedemptionPercent":    35,           // правило карты
    "totalSumRub":             2500,
    "requestedBonus":          1000,
    "effectiveBonus":          875,          // 35% от 2500 = 875, обрезано
    "wasClipped":              true,         // да, обрезано (1000 → 875)
    "cardBalanceAfterPreview": 375,
    "amountToPayRub":          1625.00       // = 2500 - 875
  }
}

UX-подсказка

Покажите юзеру слайдер «бонусы к списанию» с максимумом = min(balance, totalSum * percent / 100). На каждое изменение слайдера — quote-запрос (либо считайте локально, формула та же).

11.2 POST /web/orders service key user JWT

Создаёт заказ + атомарно списывает бонусы. Дёргается с Bitrix-бэка (нужен service key), но обязательно с пробросом user JWT — иначе нельзя проверить, что карта принадлежит этому юзеру.

Idempotency

Параметр externalId — идентификатор заказа в Bitrix. Повторный запрос с тем же externalId вернёт "duplicate": true и НЕ повторит списание. Рекомендуемый формат: "bitrix:<ORDER_ID>".

Request

POST /api/v1/web/orders
X-Service-Key: <WEB_SERVICE_API_KEY>
Authorization: Bearer <user accessToken>
Content-Type: application/json

{
  "externalId":   "bitrix:200410",
  "userCardId":   "ckxyz-abc123",
  "totalSumRub":  2500.00,
  "bonusAmount":  875,                    // итог из /quote
  "cityCode":     "khabarovsk",           // опционально, иначе берётся cityCode карты
  "shopName":     "5lb.pro web",          // опционально, попадает в CRM
  "orderMeta": {                          // опционально, произвольный JSON
    "bitrixOrderId":  200410,
    "deliveryType":   "courier",
    "courierAddress": "Хабаровск, ул. ..."
  }
}

Response 200 — обычное создание

{
  "status": "success",
  "data": {
    "orderId":            "01HJZA1B...",     // наш внутренний id (purchases.id)
    "externalId":         "bitrix:200410",
    "userId":             "ckxyz-user1",
    "userCardId":         "ckxyz-abc123",
    "cardCode":           "BASE",
    "totalSumRub":        2500.0,
    "bonusDeducted":      875,
    "wasClipped":         false,             // у юзера запрос совпал с лимитом
    "cardBalanceAfter":   375,
    "cityCode":           "khabarovsk",
    "status":             "DEDUCTED",
    "createdAt":          "2026-05-20T13:01:42.000Z"
  }
}

Response 200 — повторный запрос с тем же externalId

{
  "status": "success",
  "data": {
    "orderId":            "01HJZA1B...",
    "externalId":         "bitrix:200410",
    "duplicate":          true,
    "bonusDeducted":      875,
    "status":             "DEDUCTED",
    "cardBalanceAfter":   375,
    ...
  }
}

400INVALID_TOTAL_SUMtotalSumRub ≤ 0 или не число.

12. Полная схема интеграции Bitrix

Реалистичный сценарий: юзер кликает «личный кабинет» на сайте 5lb.pro, авторизуется, заходит в корзину, оформляет заказ со списанием бонусов, потом отменяет.

12.1 Архитектура

┌─────────────┐     CORS      ┌──────────────────┐
│  Браузер    │ ────────────▶ │ Bitrix front-JS  │
│ (5lb.pro)   │               │  (вход, профиль) │
└─────────────┘               └────────┬─────────┘
       ▲                               │ CORS
       │ HttpOnly cookie               │ Authorization: Bearer ...
       │                               ▼

       │                       │  5LB API (Rust)       │
       │                       │  mobile.5lb.pro       │
       │                       │  /api/v1/auth/...     │
       │                       │  /api/v1/bonuses/...  │
       │                       │  /api/v1/web/orders   │
       │                       └──────────────────────┘
       │                               ▲
       │                               │ X-Service-Key + Bearer
       │ Order POST (form)             │
       ▼                               │

└──────────────────┘

12.2 Пошаговый flow

  1. Вход. Юзер на сайте кликает «Войти», получает форму с SMS / Email / Telegram / VK / Apple. Фронт делает /auth/<channel>/..., бэк выдаёт accessToken + (опц.) refreshToken.
  2. Хранение токена. Бэк ставит Set-Cookie: accessToken=...; HttpOnly; Secure; SameSite=Lax; Domain=.5lb.pro при auth-ответе. Фронт ничего не пишет в localStorage — токены живут в cookie. Все последующие запросы идут с этой cookie автоматически.
  3. Шапка кабинета. Фронт дёргает GET /auth/me → имя, бонусы, аватар.
  4. Страница профиля. GET /profile/settings + PATCH /profile на сохранение.
  5. Бонусные карты. На странице «Мои карты» — GET /bonuses/cards. Если карт несколько, юзер видит баланс по каждой.
  6. Checkout. Юзер собрал корзину на сайте Bitrix (это происходит чисто на стороне Bitrix, без участия 5LB-бэка). На странице checkout фронт показывает блок «Списать бонусы»:
    • Подгружает карты юзера → юзер выбирает одну → фронт знает userCardId и баланс.
    • Слайдер «сколько списать», максимум = min(balance, totalSum * maxRedemptionPercent / 100).
    • На каждое изменение слайдера — POST /web/orders/quote → отрисовать amountToPayRub.
  7. Submit заказа. Юзер жмёт «Оплатить». Фронт сабмитит форму Bitrix (стандартный путь Bitrix). PHP-бэк Bitrix:
    • Создаёт заказ в своей БД, получает $orderId.
    • Дёргает наш бэк (см. PHP-пример ниже) с externalId = "bitrix:$orderId", userCardId, totalSumRub, bonusAmount. Передаёт Authorization юзера из cookie.
    • Получает bonusDeducted и cardBalanceAfter — записывает в свой заказ для отчётности.
    • Если бэк вернул 4xx/5xx — Bitrix отменяет создание заказа (или хранит pending-flag) и показывает юзеру ошибку.
  8. Отмена заказа. Если заказ отменили (юзер, менеджер, оплата не прошла) — Bitrix вызывает POST /web/orders/<externalId>/refund с тем же service-key. Бонусы возвращаются на карту мгновенно. Идемпотентно — можно вызвать несколько раз.
  9. Сверка. Bitrix-бэк периодически (раз в час) дёргает GET /web/orders/<externalId> для своих ещё не закрытых заказов, чтобы проверить статус (DEDUCTED / REFUNDED) — детектирует расхождения.

12.3 Bitrix PHP — пример create_order

<?php
// $userAccessToken взят из cookie 'accessToken'
// $userCardId, $bonusAmount, $totalSum — пришли с фронта (валидируем!)
// $orderId — id заказа в Bitrix

$payload = [
    'externalId'   => "bitrix:{$orderId}",
    'userCardId'   => $userCardId,
    'totalSumRub'  => (float)$totalSum,
    'bonusAmount'  => (int)$bonusAmount,
    'shopName'     => '5lb.pro web',
    'orderMeta'    => [
        'bitrixOrderId' => $orderId,
    ],
];

$ch = curl_init('https://mobile.5lb.pro/api/v1/web/orders');
curl_setopt_array($ch, [
    CURLOPT_POST           => true,
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_TIMEOUT        => 10,
    CURLOPT_HTTPHEADER     => [
        'Content-Type: application/json',
        'X-Service-Key: ' . WEB_SERVICE_API_KEY,        // лежит в config'е, НЕ в репо
        'Authorization: Bearer ' . $userAccessToken,
    ],
    CURLOPT_POSTFIELDS     => json_encode($payload, JSON_UNESCAPED_UNICODE),
]);
$body = curl_exec($ch);
$http = curl_getinfo($ch, CURLINFO_HTTP_CODE);
curl_close($ch);

if ($http < 200 || $http >= 300) {
    error_log("5LB deduct failed: HTTP $http, body=$body");
    throw new Exception('Не удалось списать бонусы');
}

$resp = json_decode($body, true);
$deducted   = $resp['data']['bonusDeducted'];
$balance    = $resp['data']['cardBalanceAfter'];
$duplicate  = $resp['data']['duplicate'] ?? false;

// Сохраняем в карточку заказа для отчётности
$order->setField('5LB_BONUS_DEDUCTED', $deducted);
$order->setField('5LB_PURCHASE_ID',    $resp['data']['orderId']);

14. Безопасность — чек-лист для разработчика

Критично. Любая утечка WEB_SERVICE_API_KEY = возможность злоумышленнику обнулять чужие бонусные балансы (или, хуже, делать refund на чужие карты). Обращайтесь с ним как с паролем от БД.

Что должен сделать Bitrix-разработчик

  1. WEB_SERVICE_API_KEY — хранить в bitrix/.settings.php или ENV-переменной хостинга. Никогда не коммитить в git, не выводить на фронт, не логировать.
  2. Все запросы к 5LB API делать ТОЛЬКО с PHP-бэка Bitrix (server-to-server). Браузер юзера должен ходить только на mobile.5lb.pro/api/v1/auth/..., /profile/..., /bonuses/..., /web/orders/quote — там X-Service-Key не используется, защита — только user JWT.
  3. Cookies: для домена 5lb.pro запросить у 5LB-команды настройку Set-Cookie Domain=.5lb.pro; SameSite=Lax — тогда фронт автоматически шлёт accessToken с каждым запросом, и не надо хранить JWT в localStorage (защищает от XSS).
  4. CSRF: cookie-режим уязвим к CSRF на мутирующих запросах. Защита — либо стандартный Bitrix CSRF-токен в формах checkout (рекомендуется), либо проверка Origin-заголовка на стороне Bitrix-бэка перед обращением к 5LB. На стороне 5LB API CORS-allowlist уже отсекает левые origin'ы.
  5. Валидация на стороне Bitrix. Перед вызовом /web/orders:
    • Проверьте, что bonusAmount с фронта ≤ maxRedemptionPercent * totalSum / 100. Бэк всё равно обрежет, но лучше отдать юзеру вразумительную ошибку до запроса.
    • Проверьте, что totalSumRub совпадает с реальной суммой корзины (юзер не отредактировал её JS-консолью).
    • Проверьте, что userCardId присутствует в списке карт юзера, полученном с /bonuses/cards.
  6. Retry-логика. Сеть может моргнуть в момент POST /web/orders и вы не узнаете, прошёл ли запрос. Решение: всегда повторите POST с тем же externalId — наш бэк идемпотентен и вернёт duplicate=true вместо двойного списания. Через 3 неудачные попытки — оповестить менеджера.
  7. Refund: всегда идемпотентен. Можно запускать из очереди (Bitrix events) и не бояться повторов.
  8. Логирование. Записывайте в Bitrix-лог каждый ответ от 5LB API: orderId, bonusDeducted, HTTP-статус. При расхождении балансов будет проще разобраться.
  9. HTTPS only. Все запросы только на https://mobile.5lb.pro. http:// отдаст 301, но лучше не полагаться на редиректы.

Что обеспечивает 5LB-бэк

  • JWT-токены подписаны HS256 (32+ байта секрет), TTL 15 минут для refresh-флоу или permanent для phone/email OTP.
  • Refresh-токены ротируются (старый сразу инвалидируется).
  • SMS/Email OTP — 4/6 цифр, bcrypt-hash в БД, constant-time сравнение, до 5 попыток, TTL 10 минут.
  • X-Service-Key сравнивается через subtle::ConstantTimeEq — нельзя выяснить ключ по timing.
  • Все запросы к БД — параметризированные (sqlx::query! макрос), SQL-инъекции невозможны.
  • Списание / refund — единая транзакция Postgres, атомарно. Race-condition'ы с одной картой защищены row-level lock'ом в UPDATE ... RETURNING.
  • Идемпотентность externalId — UNIQUE-индекс на purchases.externalId + retry в коде.
  • CORS allowlist — только явные origin'ы (5LB-команда добавит ваш домен).
  • Rate limits: 5/мин на /auth/refresh per IP, OTP-лимиты per phone/email.

Документ описывает контракт API на 2026-05-20. Все изменения формата (новые поля — backwards-compatible; удаление полей — major-bump) согласовываются с интегратором заранее.
